# 横向串讲：从「映射」看 lesson 1–10

> 这不是第 N 课的答疑，而是把 1–10 横向梳理的串讲——不按课序逐课纵向讲，而是抽出贯穿所有课的逻辑，一条线讲下来。
>
> 主线（一句话）：**Agent 工程的核心一层，是在「不可信的 LLM 输出」和「开发层的软件执行」之间，建立一层受约束、可校验、能接回动作的映射。** 1–10 全在做这一件事——它是「业务 → 产品 → 开发」整条栈的最底层（§0 说清分层）。
>
> 怎么读：六节是一条因果链，每节回答上一节引出的问题——是什么 → 怎么实现 → 为什么这个顺序 → 可靠吗 → 怎么用它搭自己的 → 业界叫什么。想要结论翻每节开头的黑体句。
>
> 配套：单课细节见各课答疑（01–10、06a/b/c）。本文只讲横向的线，不复述单课。
> 代码锚点写成 `文件:行号`，均按当前仓库核对。

## 0 总论：Agent 工程在做一件什么事

先把"真实世界需求"和"软件执行"分清楚——它们不是一层。从物理世界的业务诉求，到 LLM 能对接的东西，中间隔着多层：

```
业务层（物理世界 · 真实需求）      用户要退款 / 医生要分诊 / 工厂要排产
        ↕  映射①：业务 → 产品（产品设计）
产品层（功能 / 流程 / 规则定义）    退款流程、状态流转、交互规则
        ↕  映射②：产品 → 开发（软件设计）
开发层（软件世界 · 函数/数据/API）   算数、查库、改状态、发请求
        ↕  映射③：开发 ↔ LLM          ← 本课 1–10 只做这一层
LLM 层（概率性文本输出）
```

**课程 1–10 建的，是最底下的映射③——连接「LLM 输出」和「开发层的软件执行」，不是连接真实业务。** 业务层、产品层（映射①②）是 Agent 产品设计的工作，不在这门课范围内。下文说的"映射"，除非特别说明，都指映射③。

看清映射③这一层的两端为什么不匹配：

- 一端是**开发层的软件执行**：要可靠的数据、要触发确定的软件动作（算数、查库、改状态、发请求）。
- 另一端是 **LLM 的输出**：自由文本、概率性、会跑偏、会幻觉，**不可信**。

映射③就是在这两端之间建立一层连接。但它不是简单的"把需求翻译成 prompt"——那样会漏掉两个关键词：

- **不可信**：LLM 输出不能直接用，必须解析、校验、处理失败。lesson3 的原话："永远不要直接信任 LLM 输出。" 这一层的一半重量在校验和重试上。去掉它，就退回成普通函数调用了。
- **执行**：映射的下游不是停在"拿到结构化数据"，而是要**接回开发层的软件执行**。lesson4 之后每一课多出来的，都是把 LLM 的结构化决定接到软件动作（调函数、改状态、发请求）。lesson4 文档原话："能动性从这里开始。"

所以这层映射在代码上做三件事，对应每课的三个可变部分：

| 这层映射做什么 | 代码上是什么 | 本文叫它 |
|---|---|---|
| 约束 LLM 输出的形状 | 给定一个 schema，约束模型按它输出 | **schema** |
| 验证输出是否可信 | 解析 + 检查 + 失败重试 | **校验** |
| 把通过的输出接回软件执行 | 路由 / 调工具 / 改状态 / 存记忆 | **校验后动作** |

这就是全文的主线。下面六节，§1–§4 全部基于本项目 1–10 的代码（已知事实），§5 讲怎么用它们搭自己的 Agent，§6 才把这些标到工业界的坐标上（外部对照）。

**核心立场：没有魔法。** 所谓决策、规划、记忆、原子动作，拆开看都是上面这三件事的不同组合，没有任何隐藏的推理。

## 1 机制：1–10 是同一套模板，只在三处可变项不同〔本项目〕

**从 lesson3 起，几乎每个能力底层都是同一套代码模板；逐课的全部差异，只落在 schema、校验、校验后动作这三处可变项上。**

公共模板（每课都有，一行不差）：

```
prompt  →  temperature=0.0  →  for attempt in range(3)  →  extract_json_from_text  →  校验
```

把单次映射调用的课横排开，三处可变项一目了然：

| 课 | 方法（位置） | prompt 外壳 | schema | 校验 | 校验后动作 |
|:--|---|---|---|---|---|
| 03 结构化输出 | `generate_structured`(agent.py:126) | ChatML | 任意（调用方传） | `parsed is not None` | 直接返回 |
| 04 决策 | `decide`(agent.py:183) | ChatML | `{decision: 枚举}` | `"decision"在` 且 `in choices` | 取出 decision 值（返回 str） |
| 05 工具 | `request_tool`(agent.py:258) | ChatML | `{tool, arguments}` | 两个键都在 | 执行时再查白名单(tools.py:80) |
| 07 记忆 | `run_with_memory`(agent.py:443) | ChatML | `{reply, save_to_memory}` | `"reply"在` | 注入记忆 + 回存 + `increment_step` |
| 08 规划 | `create_plan`(planner.py:11) | 裸 f-string | `{steps: [...]}` | `"steps"在` 且是 list | agent 层存 `current_plan`(agent.py:549) |
| 09 原子动作 | `create_atomic_action`(planner.py:50) | 裸 f-string | `{action, inputs}` | `"action"在`(planner.py:90) | 直接返回 |
| 10 依赖图 | `create_aot_graph`(planner.py:96) | 裸 f-string | `{nodes:[{id,action,depends_on}]}` | 字段齐 + `depends_on`是list(planner.py:138-141) | 直接返回（执行交给 `execute_graph`） |

（lesson6 不在此表——它不是一次映射调用，而是把单次调用放进循环，见 §2。）

横排还暴露两处**只有横看才发现的真分叉**，写串讲不能只说"都一样"：

1. **prompt 外壳有两支**：`agent.py` 里的方法（03/04/05/07）套 ChatML 三段式；`planner.py` 里的（08/09/10）是裸 f-string，没有 ChatML。同一套机制，外壳分了两种。
2. **校验后动作的轻重差很多**：03/09/10 校验完直接返回（无副作用）；04 要取值、返回类型从 dict 变 str；05 执行时才查白名单；**07 副作用最多**（注入 memory + 回存 + 计步）；08 要写状态。

**两条引申线**（同一张表的两个侧面）：

- **数据形状演进**：产物 JSON 怎么从简单到复杂——`{任意}`(03) → `{decision:枚举}`(04) → `{tool,arguments}`(05) → `list[str]`(07记忆/08步骤) → `{action,inputs}`(09) → `{nodes:[...]}`图(10)。
- **显式数据存哪**：状态都是显式可见的对象，不藏在对话历史里（这是项目 PHILOSOPHY「没有魔法」的落地）——`AgentState`(steps/done) / `Memory.items` / `current_plan` / `graph`。

## 2 能动性递进：沿映射，能力逐级递增〔本项目〕

**课序按能力依赖排：04 出现能动性（从生成转为挑选），05 接回软件执行（第一次触发软件动作），06–10 再补上多步（循环）、跨轮（记忆）、任务结构（规划）。能动性这一跃发生在 04→05；06 之后是给"执行"补支撑，不再是影响力变大。**

```
01–03  只产出文本 / 数据         模型说，程序读。还没触及软件执行。
   │   01 对话 · 02 system prompt 一致性 · 03 结构化输出
   ▼
04     决策 / 路由               从有限选项里挑一个 → 程序据此走不同分支。
   │   "挑选"不是"生成"。lesson4 文档："能动性从这里开始。"
   ▼
05     工具调用                  模型发出 {tool,arguments} → 程序真的执行函数
   │   (execute_tool, tools.py:62/83；白名单 :80)。第一次触及软件执行。
   ▼
06     循环                      把单次映射调用放进 run_loop(agent.py:411)，
   │   多步自主推进；两道终止条件：done 信号 + max_steps 上限，缺一不可。
   ▼
07     记忆                      让事实跨交互留存（不靠对话历史，靠显式存储）。
   ▼
08–10  规划 → 原子动作 → 依赖图   把"多步任务"显式成数据：扁平步骤 → 带参动作
       → 带依赖的图，execute_graph(planner.py:150) 按拓扑序真执行。
```

分界点在 **lesson4**：之前模型只是"回应"，之后它开始"选择该做什么、并触发后果"。这就是为什么 04 之前是"可靠输出"，04 之后是"agent 的样子"。

**能力是层叠依赖的**（这也解释了顺序不能打乱）：

```
06 循环   复用 03/04/05 的单次调用，外面套 while
07/08     借 06 的 increment_step 记步
09        拿 08 的单个 step 当输入，转成 {action,inputs}
10        = 08 规划 + 09 原子动作 + 依赖关系（depends_on 是它唯一新增的东西）
```

## 3 可靠性边界：映射哪里严、哪里虚〔本项目〕

**这层映射的可靠性参差不齐——而且不齐是有规律的：schema 越封闭，校验越严；schema 越开放，校验越虚，甚至虚到只是个概念。**

**校验强度梯度**（按严到弱排）：

| 课 | schema | 校验 | 强度 | 为什么 |
|---|---|---|---|---|
| 04 `decide` | `{decision:枚举}` | `decision in choices`(agent.py:249) | 最严 | **封闭枚举**：值只能是白名单那几个，能逐一对 |
| 05 `request_tool` | `{tool,arguments}` | 两键 + 执行时白名单(tools.py:80) | 较严 | tool 名也有白名单可对 |
| 08 `create_plan` | `{steps:[]}` | `steps` 是 list(planner.py:44) | 中 | 只能查容器类型 |
| 03 `generate_structured` | 任意 | `parsed is not None` | 最弱 | 形状全开放，只能查"是不是 JSON" |
| 09 `create_atomic_action` | `{action,inputs}` | 只查 `"action"`(planner.py:90) | 最弱 | action 名 / inputs 全开放，无白名单可对 |

结论：**校验之虚不是作者疏漏，是 schema 开放性的固有结果。** 能把开放生成约束成封闭枚举（像 04 那样），校验才能严格——这条直接变成 §5 的设计建议。

**名实落差审计**（文档承诺 vs 代码实际，横向汇总三类）：

- **执行是占位**：`execute_plan`(agent.py:553) 只标 `executed:True`；`execute_action`(agent.py:632) 只 `return f"Executed:{action}"`。课程的"执行"大多没真做。
- **校验承诺未落地**：08/09/10 都没有环检测；09 没验文档承诺的 `inputs`；10 文档说校验会"捕获循环依赖、查依赖引用有效性"，但实际 `create_aot_graph`(planner.py:134-145) 一样没做。更甚者，10 文档里**贴的**代码（`node_ids` 版）和**实际跑的**代码（`valid_nodes` 版）不是同一份，两版都无环检测。
- **并行只在文档里**：lesson10 反复说"独立动作可以并行执行"，但 `execute_graph`(planner.py:150) 是单线程 for 循环顺序跑，没有任何并发。

这一节的意义：串讲不只是夸"设计精巧"，也要如实标出哪些是概念占位。**自己实现真实系统时，这些虚的地方就是要补的地方**（见 §5 第三步）。

## 4 范围：建映射（1–10） vs 守映射（11/12）〔本项目〕

**1–10 是建映射，这是 Agent 的主干。11/12（evals / telemetry）是守映射——它们不给 Agent 加任何运行时能力，而是保障这层映射可靠、可观测。而且它们是可选的，不是默认必须。**

证据是硬的——11/12 在代码上就是外挂：

- `agent.py` 的 import 只有 `LocalLLM / AgentState / Memory / tools / planner`（agent.py:22-27），**零引用 evals / telemetry**。
- 反向：`AgentEval.__init__(self, agent)`(evals.py:68) 把 agent 当参数从外面包住；`Telemetry`(telemetry.py) 连 agent 都不 import。

所以：删掉 11/12，agent 照跑；删掉 1–10 任一课，后面断。这就是"建映射主干 / 守映射配套"的区别。

**关键限定（修正一个常见误解）**：守映射**不是无条件必须的**，它有触发条件——**当这个 Agent 从"自己学着玩"变成"有人用、要长期维护"时**，evals（防 prompt 改动引发回归）和 telemetry（线上 trace、看成功率/延迟）才变得必要。学习阶段，映射能跑通就够了。这一点直接决定 §5 的边界。

## 5 实践：用 1–10 搭一个自己的最小 Agent〔动手〕

**目标说清楚：搭一个能跑的、自己用的最小 Agent——不是生产系统。** 所以这一节只用 1–10，不碰 11/12，也不碰成本/安全那些工业配套（理由见末尾边界线）。

设计按这四步走，全部从前面几节推导：

**第一步 · 先定映射③的两端**（决定后面一切）
- 业务需求（物理世界，设计的起点）：明确这个 Agent 要解决的真实问题。1–10 不直接处理它——需先把它经产品 / 开发层落成具体的软件动作。
- 开发层执行：明确落下来后 Agent 要触发的软件动作（调函数、查库、改状态、发请求），或确认它只产出文本 / 数据。这才是映射③直接对接的一端。

**第二步 · 选能动性层级，最低够用**（用 §2 的分级，反过度工程）
- 只要可靠回答 → 停在 lesson3，不要循环、不要工具。
- 要固定选项路由 → 加 lesson4。
- 要触发软件操作 → 加 lesson5 工具。
- 要多步自主 → 加 lesson6 循环。
- 要跨轮记忆 → 加 lesson7。
- 要先规划再执行 / 有步骤依赖 → 加 lesson8–10。
- 多搭一层都是负担。

**第三步 · 逐个 LLM 调用，设计三处可变项**（§1 + §3 的直接应用）
- schema：定下输出契约的形状。**能约束成封闭枚举就别留开放结构**——§3 证明了封闭枚举校验才严。
- 校验：验什么、重试几次、还失败怎么降级处理。**§3 里那些虚校验（无环检测、inputs 没验、占位执行）就是要补的地方**。
- 校验后动作：定下结构化决定怎么接回执行。**工具白名单就是安全边界**；不可逆动作（删数据、发消息、付款）需考虑人工确认 / dry-run。

**第四步 · 按需加状态与控制**
- 单轮任务就不要状态 / 记忆（§1 的"显式数据存哪"，但不需要的别加）。
- 有循环才要两道终止条件：`done` + `max_steps`（防无限循环、防成本失控）。
- 记忆会涨：全量注入迟早撑爆上下文窗口（lesson7 的已知问题）→ 考虑截断 / 检索。

到这，一个能跑的学习 Agent 就成立了。

**边界线（这一节本身的诚实结论）**：

> evals、telemetry，以及成本 / 安全 / 错误降级，**都不是"完整 Agent"的必需件**，而是当这个 Agent 从学习项目走向"有人用、要长期维护"时才需要加的。**学习阶段加它们，就是过度工程**——这正是本项目 PHILOSOPHY「最小可运行优先，不过度设计」的要求。

## 6 接到业界坐标：这些设计决策，业界叫什么〔本项目 → 外部〕

**前面五节全在这个项目内部自洽。现在从外部看：§5 的每一步设计决策，工业界都有对应的方法论——而 1–10 已经从第一性原理把它们各实现了一个最小原型。**

| §5 的设计动作 | 工业界方法论 | 本课的最小原型 |
|---|---|---|
| 把够用的信息组织进有限 prompt | 上下文工程（context engineering） | 各课的 prompt 组装 |
| 让事实跨轮活下来 | 记忆工程（memory engineering） | lesson7 `run_with_memory` |
| 让模型触发真实能力 | 工具调用 / 函数调用（tool / function calling） | lesson5 `request_tool` + `execute_tool` |
| 让模型在有限选项里受控选择 | 决策封装 / 路由（routing） | lesson4 `decide` |
| 让输出能被程序消费 | 结构化输出 / 约束解码 | lesson3 `generate_structured` |

**但"都是成熟方法论"要分级**，成熟度差得很远：

- **已定型**：工具调用 / 函数调用（2023 年中 OpenAI 引入 function calling，Anthropic tool use）、结构化输出（JSON mode / structured outputs）——有官方 API 级标准实现。
- **2025 才成形**：上下文工程是 2025 年年中才 settled 的提法（6 月 Tobi Lütke、Andrej Karpathy 公开推广，Anthropic 9 月才正式定义），它如今把 prompt engineering 纳为自己的子集。
- **更年轻、更不定型**：记忆工程——业界常把"对话历史、agent 状态、检索内容"直接归进 context engineering，即记忆多被视作上下文工程在时间维度上的子问题，而非独立成熟方法论。

所以这些方法论不是一堆并列的成熟技术，而是**那层映射拆出的不同子问题各自的解法**：有的偏输入侧（上下文 / 记忆），有的偏输出侧（工具 / 决策），有的偏可靠性（结构化输出）。

**核心要点：** 课程 1–10 建的是映射③——在「不可信的 LLM 输出」和「开发层的软件执行」之间，建一层受约束（schema）、可校验（校验）、能接回动作（校验后动作）的映射；它只是「业务 → 产品 → 开发」整条栈的最底层，业务层 / 产品层不在课程范围。1–10 共享同一套模板建这层映射，只在三处可变项不同（§1）；课序是能动性从"说"到"做"的递进（§2）；映射的可靠性随 schema 封闭性而定，且有大量概念占位（§3）；11/12 是可选的守映射配套（§4）；用它搭自己的学习 Agent 只需四步、不必上工程配套（§5）；而 1–10 做的这一切，正是工业界上下文工程 / 工具调用 / 决策封装等方法论的最小原型（§6）——没有魔法。
