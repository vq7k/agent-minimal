## 1. 目标

S4 只解决一件事：

`alpha` 的会话消息落 PostgreSQL，页面刷新后能按同一个 `conversation_id` 恢复历史。

本片完成后：

- 同一 `conversation_id`：能读回历史消息。
- 新 `conversation_id`：不混入旧会话。
- 每轮发送：先写 user，再调用 `alpha.chat(full_messages)`，最后写 assistant。
- 前端刷新：保留 `conversation_id`，拉取历史并恢复页面。

---

## 2. 非目标

S4 不做以下内容：

- 不做偏好抽取。
- 不做长期用户画像。
- 不做语义检索 / 向量库。
- 不做复杂会话列表。
- 不改 `alpha.chat()` 的 S1-S3 内部逻辑。

理由：S4 是 DB 底座。偏好微调放到 S5，避免一次写太多。

## 4. 数据表

只存消息，不存 profile。

```sql
CREATE TABLE IF NOT EXISTS alpha_messages (
  id BIGSERIAL PRIMARY KEY,
  conversation_id TEXT NOT NULL,
  role TEXT NOT NULL CHECK (role IN ('user', 'assistant')),
  content TEXT NOT NULL,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX IF NOT EXISTS idx_alpha_messages_conversation_id_id
ON alpha_messages (conversation_id, id);
```

## 5. 后端 API

### 5.1 读取历史

```text
GET /agents/alpha/conversations/{conversation_id}/messages
```

响应：

```json
{
  "messages": [
    {"role": "user", "content": "周六去杭州"},
    {"role": "assistant", "content": "Alpha ready."}
  ]
}
```

### 5.2 发送消息

```text
POST /agents/alpha/conversations/{conversation_id}/chat
```

请求：

```json
{
  "message": "砍到3个景点"
}
```

响应仍是现有 SSE：

```text
data: {"type":"text","delta":"..."}

data: {"type":"done"}
```

## 6. 后端编排

```text
POST /agents/alpha/conversations/{conversation_id}/chat
  ├─ message 去空格；空文本返回 400
  ├─ history = storage.list_messages(conversation_id)
  ├─ storage.append_message(conversation_id, "user", message)
  ├─ full_messages = history + 当前 user message
  ├─ stream = alpha.chat(full_messages)
  ├─ 边转发 SSE 边收集 assistant text delta
  └─ 流结束后写入 assistant 完整回复
```

关键点：

- `alpha.chat()` 仍只接收 `messages`，不直接接触 DB。
- DB 只在 `server.py` 的会话 API 编排层使用。
- assistant 只存 `text` delta 拼接后的内容，不存 `done`。

## 8. 前端设计

新增最小状态：

```text
conversationId：localStorage 保存
messages：页面展示消息
isLoadingHistory：刷新后拉历史
```

页面加载：

```text
读取 localStorage conversationId
没有则 crypto.randomUUID() 创建
GET history
有历史则展示历史；无历史则展示 Alpha ready.
```

点击 New：

```text
生成新 conversationId
清空页面消息为 Alpha ready.
不删除旧会话历史
```

验收后补充：仅“不删除旧会话历史”不够。页面暴露 `New` 后，必须同时提供旧会话入口；
否则旧会话虽在 DB，但用户不可见。修复见 [S4.1 历史会话列表修复设计](./S4.1-历史会话列表修复设计.md)。

## 9. 降级与错误

| 场景 | 处理 |
|---|---|
| 空 message | 400 `message is required` |
| 读取历史失败 | 前端显示错误，保留初始消息 |
| 写 user 失败 | 不调用 `alpha.chat()`，避免假成功 |
| assistant 流式中断 | 前端按现有错误处理；已写入 user，assistant 不写空内容 |
| DB 不可用 | 返回明确错误，不静默丢消息 |
