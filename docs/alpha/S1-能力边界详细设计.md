---

## 1. 目标

S1 只解决一件事：alpha 要知道自己**只做一日游规划相关对话**。当用户问明显无关的问题时，alpha 不应直接回答，也不应强行规划，而是礼貌说明边界并引导回出游场景。

本片完成后，alpha 仍然是一个可用聊天 agent：

- `out_of_scope`：拒答 + 引导回一日游规划。
- `qa`：继续走现有普通聊天兜底。
- `plan`：本片先不做规划，也继续走现有聊天兜底；真正行程生成留给 S2。

---

## 2. 非目标

S1 不做以下内容：

- 不做城市/日期槽位抽取。
- 不做行程生成。
- 不做偏好记忆。
- 不新增前端 UI。
- 不抽独立 `sse.py`。
- 不做真实联网评测或完整 e2e。

理由：S1 是能力边界切片，用户可见增量是“问错方向会被正确挡住”。把规划、追问、过程可见塞进来会破坏垂直切片。

---

## 3. 当前基线

现状：

- `agents/alpha/__init__.py` 自己直接创建 OpenAI client，并把模型流式输出包成 SSE `text` 事件。
- `_core/llm.py` 已有两个底座：
  - `generate_structured(prompt, schema_hint, validate, retries=3)`：结构化输出 + 校验 + 重试 + `None` 降级。
  - `stream_text(messages, system=None)`：纯文本流式输出，返回裸文本 chunk。

S1 应把 `chat()` 从“直接调 SDK”切到复用 `_core.llm`，让入口变成编排层。

## 5. 路由设计

### 5.1 分类

路由结果只允许三类：

| kind | 含义 | S1 动作 |
|---|---|---|
| `plan` | 用户有出游/一日游规划意图 | 走现有聊天兜底 |
| `qa` | 普通寒暄或与 alpha 能力相关的轻问答 | 走现有聊天兜底 |
| `out_of_scope` | 明显非出游、非 alpha 能力范围 | 拒答 + 引导 |

### 5.2 `qa` 和 `out_of_scope` 的边界

`qa` 不是“什么都答”。它只保留两类轻量对话：

- 寒暄：如“你好”“你是谁”。
- 能力说明：如“你能做什么”“怎么帮我规划”。

以下应归为 `out_of_scope`：

- 创作：写诗、写代码、写邮件。
- 泛知识问答：解释历史、数学、新闻。
- 与一日游无关的任务执行。

这个边界让 alpha 的产品感更清楚：它可以寒暄和说明能力，但不会退化成通用助手。

## 6. `intent.py` 设计

### 6.1 对外接口

```python
Kind = Literal["plan", "qa", "out_of_scope"]


def route(messages: list[dict]) -> Kind:
    ...
```

### 6.2 行为

`route()` 内部调用：

```python
llm.generate_structured(
    prompt=...,
    schema_hint='{"kind":"plan|qa|out_of_scope"}',
    validate=_validate_route,
)
```

校验函数只接受：

```python
data.get("kind") in {"plan", "qa", "out_of_scope"}
```

如果 `generate_structured()` 返回 `None`，`route()` 降级为：

```python
"qa"
```

理由：路由失败时不应误拒用户。兜 `qa` 等于走原有聊天能力，风险最小。

## 7. Prompt 设计

路由 prompt 要短，避免把分类任务变成推理长文：

```text
你是 alpha agent 的意图路由器。
alpha 只做一日游/出游规划相关对话。

请根据完整 messages 判断最后一条用户消息属于哪类：
- plan：用户想规划一日游、出游路线、景点、餐饮、预算、时间安排，或补充/修改出游需求。
- qa：寒暄，或询问 alpha 能做什么、如何使用 alpha。
- out_of_scope：明显与出游规划无关的创作、代码、通用问答、工作任务等。

只输出 JSON：{"kind":"plan|qa|out_of_scope"}
```

实现时 prompt 应把完整 `messages` 作为 JSON 文本附在后面，使用 `ensure_ascii=False`，方便中文输入保真。

## 8. `chat()` 编排设计

### 8.1 基本流程

```text
chat(messages)
  ├─ kind = intent.route(messages)
  ├─ if kind == out_of_scope:
  │    ├─ yield text(拒答文案)
  │    └─ yield done
  └─ else:
       ├─ for delta in llm.stream_text(messages, system=SYSTEM_PROMPT):
       │    └─ yield text(delta)
       └─ yield done
```

### 8.2 SSE 输出

S1 继续保持现有契约：

```json
{"type":"text","delta":"..."}
{"type":"done"}
```

不要新增 `status`，也不要改前端解析。

### 8.3 text 包装

S1 可在 `__init__.py` 内保留两个小函数：

```python
def _text(delta: str) -> str:
    return f"data: {json.dumps({'type': 'text', 'delta': delta}, ensure_ascii=False)}\n\n"


def _done() -> str:
    return 'data: {"type":"done"}\n\n'
```

这些函数足够小，等 S5 再抽到 `sse.py`。

## 9. 拒答文案

拒答文案要短，给用户下一步：

```text
我现在只专注做一日游规划，暂时不处理这类请求。你可以告诉我想去的城市、日期、同行人和偏好，我来帮你安排一天的路线。
```

不使用 LLM 生成拒答。原因：

- 文案固定，没必要多一次模型调用。
- 边界表达应稳定，避免模型绕开产品边界。

## 10. 异常与降级

### 10.1 路由结构化失败

`generate_structured()` 返回 `None`：

- `intent.route()` 返回 `qa`。
- `chat()` 走现有流式聊天。

这是“保守不误杀”的降级。

### 10.2 `stream_text()` 失败

S1 不额外吞异常。若缺 `DEEPSEEK_API_KEY` 或 SDK 调用失败，沿用当前服务错误表现。统一友好错误可以后续独立处理，不塞进 S1。

### 10.3 空 messages

如果 `messages` 为空：

- `route()` 可仍调用模型判断，但没有最后用户消息时容易不稳定。
- 推荐 `chat()` 直接走 `qa` 兜底，让模型按普通助手打招呼。

## 11. 最小回归

只测“LLM 输出接软件分支”的边界，不追求覆盖率。

### 11.1 `intent.route()`

文件：

```text
tests/test_alpha_intent.py
```

用 `monkeypatch` mock `llm.generate_structured`，覆盖：

1. `{"kind":"plan"}` → `route()` 返回 `plan`
2. `{"kind":"qa"}` → `route()` 返回 `qa`
3. `{"kind":"out_of_scope"}` → `route()` 返回 `out_of_scope`
4. `None` → `route()` 返回 `qa`

不测 prompt 文案细节，只测分类结果和降级。

### 11.2 `chat()` 分支

可在同一个测试文件或新增 `tests/test_alpha_chat.py`：

1. mock `intent.route` 返回 `out_of_scope`，断言输出包含拒答 text 和 done。
2. mock `intent.route` 返回 `qa`，mock `llm.stream_text` 返回 `["你好"]`，断言输出 text + done。

不测试真实 OpenAI SDK，不联网。

## 12. 人工验收标准

本地运行服务后，在 alpha 页面或 API 中验证：

| 输入 | 期望 |
|---|---|
| `帮我写首诗` | 礼貌拒答，并引导提供城市/日期/偏好 |
| `你好` | 正常聊天 |
| `你能做什么` | 正常说明能力 |
| `周六去杭州玩一天` | S1 阶段仍可能只是普通聊天，不要求行程表 |

最后一条不要求行程表，是为了严格守住 S1 范围；S2 才验收行程生成。

## 14. 设计取舍

| 方案 | 结论 | 理由 |
|---|---|---|
| `intent.py` 独立模块 | 采用 | S1 的核心就是路由，模块名与功能对应，读任务拆分能找到代码 |
| 路由内联在 `__init__.py` | 不采用 | 初期更少文件，但会让入口重新变胖 |
| `out_of_scope` 用 LLM 生成拒答 | 不采用 | 稳定边界比文案灵活更重要 |
| 路由失败兜 `out_of_scope` | 不采用 | 容易误杀正常用户输入 |
| 现在抽 `sse.py` | 不采用 | 重复不足，等 S5 有 status 再抽 |

---

## 15. 完成标准

S1 完成时应满足：

- `帮我写首诗` 不再被当通用助手回答。
- `你好` 仍可正常聊天。
- `plan/qa/out_of_scope` 三分支有最小回归保护。
- 不改前端，不引入新依赖。
